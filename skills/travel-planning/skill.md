# Travel Planning Skill

帮助用户规划完整的旅行行程，生成结构化的行程安排。

---

## When to use
- 用户询问旅行规划、行程安排
- 用户提供目的地和时间，需要制定计划
- 用户需要景点、交通、住宿推荐

## When NOT to use
- 用户只询问单一信息（如只问天气）
- 用户在闲聊、不涉及旅行
- 用户明确表示只是查询，不需要规划

---

## Goal

根据用户的需求，生成一份完整的旅行行程安排，包括：
- 每日行程安排
- 交通方式建议
- 住宿推荐
- 景点介绍
- 费用预估

---

## Steps

执行此 skill 时，按以下步骤进行：

### Step 1: 收集必要信息

确认以下信息是否齐全，如有缺失则向用户询问：

- **目的地**：用户想去哪里
- **出发地**：用户从哪里出发
- **出行日期**：出发和返回日期
- **出行人数**：成人、儿童数量
- **预算范围**：经济/舒适/豪华（可选）
- **偏好**：自然风光/人文历史/美食购物/休闲度假（可选）

### Step 2: 验证和查询目的地信息

1. **验证用户输入**：
   - 检查出发日期 < 返回日期
   - 确认目的地是真实存在的城市
   - 如有问题，立即反馈用户并要求修正

2. **查询目的地信息**（如工具可用）：
   - 尝试查询天气预报（如不可用，使用季节性通用建议）
   - 获取目的地的基本信息（地理位置、主要景点、交通枢纽）
   - 如工具不可用，基于知识库提供建议

### Step 3: 规划每日行程

根据出行天数和用户偏好，安排每日行程：

1. **景点选择原则**：
   - 根据用户偏好筛选景点（自然风光/人文历史/美食购物/休闲度假）
   - 优先选择知名度高、评价好的景点
   - 考虑景点开放时间和游览时长

2. **行程安排原则**：
   - 每天安排 2-4 个景点，避免过于紧凑
   - 相近景点安排在同一天，减少往返
   - 预留用餐和休息时间（每天至少1小时午休）
   - 第一天考虑到达时间，可能只安排1-2个景点
   - 最后一天预留返程时间，不安排远距离景点

3. **行程示例**：
   - 第一天：抵达 → 入住 → 附近景点或休息
   - 中间天：主要景点游览，合理分配
   - 最后一天：轻松景点或购物 → 返程

### Step 4: 推荐交通方式

综合考虑多个因素：

- **距离**：作为主要参考（> 1000km优先飞机，300-1000km考虑高铁，< 500km可自驾）
- **时间偏好**：用户是否赶时间（赶时间优先飞机/高铁）
- **预算**：经济预算优先大巴/普通火车，豪华预算可飞机
- **出行人数**：多人出行可考虑自驾或包车
- **实际网络**：中国主要城市间优先推荐高铁（性价比高）

### Step 5: 推荐住宿区域

根据景点分布和用户需求推荐住宿区域：

1. **位置选择**：
   - 优先选择交通便利的区域（靠近地铁/公交枢纽）
   - 靠近主要景点集中的区域，减少通勤时间
   - 考虑周边配套设施（餐厅、便利店、药店）

2. **预算匹配**：
   - 经济预算：推荐青年旅舍、快捷酒店集中区
   - 舒适预算：推荐商务酒店、连锁酒店集中区
   - 豪华预算：推荐五星级酒店、度假村区域

3. **房型建议**：
   - 2人：大床房或双床房
   - 3-4人：家庭房或两间房
   - 5人以上：考虑公寓式酒店或民宿

### Step 6: 费用预估

根据以下标准估算费用（均为参考值，以实际预订为准）：

1. **交通费用**：
   - 飞机：根据距离估算（短途500-1500元，中途1000-2500元，长途1500-4000元）
   - 高铁：约0.4-0.5元/公里（如北京-上海约550元/人）
   - 自驾：约0.8-1.2元/公里（含油费、过路费）

2. **住宿费用**（每晚）：
   - 经济：150-300元
   - 舒适：300-600元
   - 豪华：600元以上

3. **景点门票**：根据目的地常见景点价格估算（一般100-300元/天）

4. **餐饮费用**（每人每天）：
   - 经济：80-150元
   - 舒适：150-300元
   - 豪华：300元以上

### Step 7: 生成行程单

整合以上信息，生成结构化的行程单。

## Parameters

本 skill 通过对话收集参数。用户可能提供的信息及验证规则：

| 参数 | 必需 | 说明 | 验证规则 |
|------|------|------|--------|
| `destination` | ✓ | 目的地城市 | 必须是真实存在的城市 |
| `start_date` | ✓ | 出发日期 | 格式：YYYY-MM-DD，不能早于今天 |
| `end_date` | ✓ | 返回日期 | 必须晚于出发日期 |
| `origin` | ✗ | 出发地城市 | 如未提供，可询问或推断（如用户常驻地） |
| `travelers` | ✗ | 出行人数 | 默认2人，需要时询问 |
| `budget` | ✗ | 预算偏好 | 经济/舒适/豪华，影响交通和住宿推荐 |
| `preferences` | ✗ | 旅行偏好 | 自然风光/人文历史/美食购物/休闲度假，影响景点选择 |

## Example

**用户输入**：
> 帮我规划一下下周从北京去上海的旅行，大概玩3天，2个人

**执行过程**：
1. 确认信息：出发地北京，目的地上海，3天2晚，2人
2. 查询上海天气
3. 规划行程：外滩、东方明珠、豫园、田子坊等
4. 推荐交通：高铁（4.5小时，约550元/人）
5. 推荐住宿：外滩/南京路附近
6. 生成完整行程单

**输出示例**：
```

- 目的地：上海
```

## Notes

- **信息收集**：如果用户信息不完整，先询问再规划。优先级：目的地 > 日期 > 出发地（可推断）
- **可行性检查**：考虑景点开放时间、交通衔接、实际游览时长
- **费用预估**：为参考值，实际以预订时为准。如无法获取实时价格，给出历史参考价格
- **灵活调整**：根据用户反馈随时调整行程，不必重新生成整个计划
- **工具降级**：如天气/交通工具不可用，基于知识库提供建议，不中断规划流程

---

## Error Handling

遇到以下情况时的处理方式：

| 错误类型 | 处理方式 |
|---------|---------|
| 日期矛盾（出发日期 > 返回日期） | 提示用户："返回日期应该晚于出发日期，请确认您的日期" |
| 日期已过期 | 提示用户："出发日期已过期，请提供未来的日期" |
| 目的地不存在/无法识别 | 提示用户："无法识别该目的地，请确认城市名称是否正确" |
| 出行天数过长（> 15天） | 提示用户："行程较长，建议分段规划或指定重点关注区域" |
| 用户中途修改需求 | 根据修改内容调整相应部分，无需重新收集所有信息 |

## Multi-turn Conversation

用户可能会在生成行程后继续交互，常见场景：

1. **修改行程**：
   - 用户："第二天太累了，少安排点景点"
   - 处理：调整第二天行程为1-2个景点，增加休息时间

2. **增加信息**：
   - 用户："我们有个小孩，5岁"
   - 处理：重新考虑景点是否适合儿童，调整餐饮推荐

3. **特定要求**：
   - 用户："不想去东方明珠，换个景点"
   - 处理：替换为其他景点（如上海中心大厦、金茂大厦）

4. **追问细节**：
   - 用户："酒店有推荐的吗？"
   - 处理：提供具体酒店名称和预订链接建议
